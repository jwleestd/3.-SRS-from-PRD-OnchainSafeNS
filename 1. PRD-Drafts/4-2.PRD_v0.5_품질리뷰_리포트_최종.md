## Part B. 잔존 결함 상세 (7건 — 전량 Low/Minimal)

> 3차 리뷰에서 **Medium 이상 결함은 0건**이다. 아래는 문서 완성도를 높이기 위한 폴리시 수준 개선 항목이다.

### B-1. §5.2 FUNC-024 "수집 주기 준수" — 주기 미명시 (Low)

**현행:** AC가 "수집 주기 준수"로, 구체적인 주기(15분? 1시간? 소스별 상이?)가 AC에 명시되지 않았다. CON-5(외부 소스 API ≤ 15분 갱신)는 §10에 있으나 AC 내에서 참조되지 않는다.

**수정 방향:** AC를 "수집 주기 ≤ 15분 준수 (CON-5 참조). FRAUD_INTELLIGENCE_SOURCE.collection_schedule 기준"으로 구체화.

### B-2. §5.2 FUNC-051~052 번들 (Low)

**현행:** 2건이 단일 행으로 묶여 AC "정확도 100%"이다. "품질 검증"과 "승인 대기열"은 다른 기능이므로 테스트 케이스를 분리하기 어렵다.

**수정 방향:** 2건으로 분리. FUNC-051 "Staging 품질 검증: cross_source_count ≥ 2 → quality_check_status='passed'" / FUNC-052 "승인 대기열: quality_check_status='passed' 건만 approval_status='pending_approval'로 노출".

### B-3. §4.4 BM-1 테스트셋 라벨링 방법론 (Low)

**현행:** "사기 주소 1,000건 + 정상 주소 9,000건" 테스트셋의 라벨링 기준(누가, 어떤 근거로 사기/정상을 판정하는가)이 정의되지 않았다.

**수정 방향:** BM-1에 "Ground Truth: Chainalysis KYT 공개 리포트 + OFAC SDN 리스트에서 확인된 주소를 '사기'로 라벨링, 6개월 이상 정상 거래 이력이 있고 어떤 소스에도 미등록인 주소를 '정상'으로 라벨링" 단서를 추가.

### B-4. §5.2 Phase-1 Must AC 비-GWT (잔존 — Minimal)

**현행:** Phase-1 Must 항목 ~12건이 수치 나열형 AC(예: "≤ 2초", "정확도 100%")이다.

**판정:** v0.4 리뷰에서 "Phase-1 진입 시 전량 GWT 전환" 정책이 수립되었으므로 설계상 유보(By Design). 현 시점에서 수정 불요.

### B-5. §5.2~5.4 AC 공란 5건 (잔존 — Minimal)

**현행:** FUNC-025~027(Should), FUNC-036(Should), FUNC-042(Should), FUNC-028~030(Should). 전량 Should/Could 등급.

**판정:** 구현 스프린트 착수 시 AC 작성 정책 수립 완료. 현 시점 수정 불요.

### B-6. §7~12 참조 방식 (Minimal — 구조)

**현행:** "v0.4 원문 유지"로 §7~12를 외부 참조한다. 단독 열람 시 불편하나, 측정 가능성 결함은 아님.

**판정:** 차기 버전(v0.6 이상)에서 인라인 포함 또는 별도 Appendix 문서 링크로 개선. 현 시점 수정 불요.

### B-7. §5.2 FUNC-031 "≤ 30일" 범위 모호 (Low)

**현행:** "소스 정책 변경 감지 + 자동 전환 ≤ 30일" — 감지~전환 완료까지 30일인지, 감지 후 전환만 30일인지 불명확.

**수정 방향:** AC를 "소스 정책 변경 공지 수신(CON-7) 후 대체 소스 전환 완료까지 ≤ 30일. 전환 중 커버리지 감소 ≤ 5% (H8 참조)"로 구체화.

## Part E. 최종 판정

**✅ PRD v0.5는 Phase-0 구현 착수에 충분한 품질을 확보하였다.**

Phase-0 범위(19 REQ + 10 NFR)는 전량 GWT AC + 모니터링 + 위반 조치 + 검증 계획이 완비되어 있으며, 개발자가 테스트 케이스를 직접 도출할 수 있는 수준이다. 잔존 리스크 3건은 전량 Minimal이며 Phase-1 진입 시 순차 해소된다.

추가 리뷰 라운드는 수확 체감(Diminishing Returns) 구간에 진입하였으므로, 다음 단계는 **구현 착수 → E2E 테스트 작성 → Phase-0 Validation** 순서를 권장한다.
