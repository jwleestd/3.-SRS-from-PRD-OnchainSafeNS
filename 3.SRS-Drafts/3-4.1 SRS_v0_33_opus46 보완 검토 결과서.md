# SRS v0.33 보완 검토 결과서

**Document ID:** SRS-REVIEW-V033  
**작성일:** 2026-05-03  
**대상 문서:** SRS v0.32 (3-3.2 SRS_v0_32_opus46.md)  
**검토 기준:** Mastercard Crypto Credential(MCC) 유사 서비스 대비 송수신자 신뢰성 강화 전략 수립  

---

## 1. 검토 배경 및 목적

본 검토는 Mastercard Crypto Credential(이하 MCC)이 제공하는 신뢰 강화 메커니즘을 분석하고, 현재 SRS v0.32의 On-Chain Fraud Shield Platform이 채택하지 않고 있는 핵심 기능을 식별하여 보완 전략을 수립하는 것을 목적으로 한다.

검토 범위는 요청된 3가지 전략 영역에 한정한다.

| # | 전략 영역 | 검토 코드 |
|---|---|---|
| 가 | 송수신 사전 검증 | CR-V33-A |
| 나 | 자금 송금·금융 결제 시 KYC 기반 신원 검증 | CR-V33-B |
| 다 | 지원되지 않는 온체인으로 금융서비스를 이용하는 실수 예방 | CR-V33-C |

---

## 2. Mastercard Crypto Credential(MCC) 핵심 기능 분석

MCC는 2024년 5월 정식 출시 이후 유럽, 중남미, 중동, 아시아 등으로 확장 중이며, 2025년 11월 Polygon Labs·Mercuryo와 협력하여 셀프 커스터디 지갑까지 지원을 확대하였다. MCC의 핵심 신뢰 강화 기능은 다음과 같다.

### 2.1 MCC 핵심 기능 요약

| # | 기능 | 설명 |
|---|---|---|
| MCC-1 | **KYC 기반 검증된 별칭(Verified Alias)** | 사용자가 KYC 검증을 완료한 후에만 Human-Readable 별칭을 발급받으며, 별칭 자체가 신원 검증 완료를 의미 |
| MCC-2 | **송수신 사전 검증** | 송금 시작 전 수신자의 별칭 유효성, 신원 검증 상태, 지갑 호환성을 사전 확인하여 트랜잭션 진행 여부를 결정 |
| MCC-3 | **자산·체인 호환성 사전 검증** | 수신자의 지갑이 전송하려는 특정 디지털 자산과 블록체인을 지원하는지 확인하고, 미지원 시 송금자에게 통보 후 트랜잭션을 차단 |
| MCC-4 | **Soulbound Token(SBT)** | Polygon 블록체인 위에 발행되는 양도 불가 토큰으로, 해당 지갑이 검증된 사용자에 속함을 온체인으로 증명 |
| MCC-5 | **Travel Rule 준수** | 국경 간 트랜잭션 시 FATF 권고에 따른 송수신자 정보 교환을 자동화 |
| MCC-6 | **신뢰된 네트워크 거버넌스** | 모든 참여 거래소·제공자에게 최소 거버넌스 프레임워크를 적용하여 네트워크 전체의 신뢰 수준을 표준화 |

### 2.2 MCC vs 현 SRS v0.32 Gap 분석

| 비교 항목 | MCC | SRS v0.32 (현재) | Gap 수준 |
|---|---|---|---|
| 별칭/이름 등록 시 KYC 신원 검증 | KYC 완료 후 별칭 발급 (필수) | Safe-Name 등록 시 지갑 서명만 요구, KYC 없음 | **Critical Gap** |
| 송금 전 수신자 신원 검증 상태 확인 | 별칭 유효성 + 신원 검증 상태 사전 확인 | 사기 DB 교차 조회만 수행, 수신자 신원 검증 상태 미확인 | **Critical Gap** |
| 자산·체인 호환성 사전 검증 | 수신 지갑의 자산·체인 지원 여부 확인 후 비호환 시 차단 | 미지원 체인 안내(REQ-FUNC-005)는 있으나, 수신자 지갑의 자산·체인 호환성 사전 검증 부재 | **Major Gap** |
| 온체인 검증 증명 (SBT) | Soulbound Token으로 검증 상태 온체인 공개 | 해당 기능 없음 | **Minor Gap** (MVP 이후 가능) |
| Travel Rule 준수 자동화 | 내장 | 해당 기능 없음 (OS-8 삭제와 연계) | **Out-of-Scope** (향후) |
| 네트워크 거버넌스 프레임워크 | 모든 참여자에 표준 적용 | VASP 개별 API Key 인증만 존재 | **Minor Gap** |

## 3. 전략별 상세 Gap 분석 및 보완 방향

### 3.1 CR-V33-A: 송수신 사전 검증

#### 현재 상태 (As-Is)

SRS v0.32의 Safe-Name 기반 송금 플로우(§3.4.4)에서는 송금자가 수신자 이름을 입력하면 오프체인 레지스트리에서 주소를 리졸브한 후 사기 DB 교차 조회를 수행한다. 그러나 다음 항목은 검증되지 않는다.

- 수신자의 Safe-Name 소유자가 KYC 검증된 사용자인지 여부
- 수신자 지갑이 해당 자산·체인을 수신 가능한지 여부
- 송금자 자신의 KYC 검증 상태가 유효한지 여부

#### 보완 방향

| 보완 ID | 보완 항목 | 영향 범위 |
|---|---|---|
| CR-V33-A-1 | Safe-Name 리졸브 응답에 **수신자 KYC 검증 등급(Verification Tier)** 정보를 포함하고, 미검증 수신자에게는 경고를 표시한 후 송금자의 명시적 확인을 요구 | REQ-FUNC-015, Safe-Name Resolve API, SAFE_NAME 엔터티 |
| CR-V33-A-2 | 송금 전 **수신자 지갑의 자산·체인 호환성(Asset-Chain Compatibility)**을 검증하여, 비호환 시 트랜잭션을 차단하고 안내 메시지를 표시 | REQ-FUNC-015/016 신규 AC 추가, 신규 API 필드 |
| CR-V33-A-3 | 일정 금액 이상(예: $1,000) 트랜잭션 시 **송금자 KYC 검증 등급 확인 + 수신자 KYC 교차 확인**을 수행하는 Enhanced Verification 단계 신설 | 신규 REQ-FUNC, Threshold Policy 설정 |

#### 신규/수정 요구사항 도출

| ID | 요구사항 요약 | Priority |
|---|---|---|
| **REQ-FUNC-053** | 송금 전 수신자 사전 검증(Pre-Transfer Recipient Verification) — 수신자의 KYC 검증 등급·자산 호환성·사기 DB 상태를 종합 확인 | Must |
| **REQ-FUNC-054** | 고액 송금 시 Enhanced Verification — 송수신 양측 KYC 등급 교차 확인 | Should |

### 3.2 CR-V33-B: KYC 기반 신원 검증

#### 현재 상태 (As-Is)

SRS v0.32에서 Safe-Name 등록(REQ-FUNC-014)은 지갑 주소 소유권 서명만을 요구하며, 등록자의 실제 신원에 대한 KYC 검증은 수행하지 않는다. 이로 인해 다음 문제가 발생한다.

- 사기범이 Safe-Name을 등록하여 신뢰할 수 있는 이름으로 위장 가능
- 수신자 입장에서 송금자가 검증된 사용자인지 판단 불가
- 금융 규제 당국의 자금세탁방지(AML)/고객확인(KYC) 요구사항 미충족

#### MCC와의 핵심 차이점

MCC는 별칭 발급 자체가 KYC 완료를 전제로 하므로, 별칭을 가진 사용자는 자동으로 신원 검증된 상태가 된다. 반면, 현 SRS의 Safe-Name은 이름 등록과 신원 검증이 분리되어 있어, 이름만으로는 상대방의 신뢰성을 판단할 수 없다.

#### 보완 방향

| 보완 ID | 보완 항목 | 영향 범위 |
|---|---|---|
| CR-V33-B-1 | Safe-Name에 **KYC Verification Tier(검증 등급)** 체계 도입: Tier-0(미검증), Tier-1(기본 검증: 이메일·전화번호), Tier-2(신원 검증: ID 문서·생체 인증), Tier-3(기관 검증: 법인 KYB) | SAFE_NAME 엔터티 확장, Safe-Name Register API 수정 |
| CR-V33-B-2 | Safe-Name 등록 시 최소 **Tier-1 검증(이메일·전화번호 인증)**을 필수 조건으로 설정하고, Tier-2 이상 검증 시 "Verified" 배지를 부여 | REQ-FUNC-014 AC 수정, 신규 KYC 검증 서비스 컴포넌트 |
| CR-V33-B-3 | 금융 목적 송금 시 송수신자 모두 **최소 Tier-1 검증** 필수, $1,000 이상 송금 시 **Tier-2 검증** 필수 정책 도입 | 신규 정책 요구사항, Threshold 설정 |
| CR-V33-B-4 | KYC 검증 상태를 Safe-Name 리졸브 결과에 포함하여, 송금자가 수신자의 검증 등급을 확인 가능 | Safe-Name Resolve API 응답 확장 |

#### 신규/수정 요구사항 도출

| ID | 요구사항 요약 | Priority |
|---|---|---|
| **REQ-FUNC-055** | Safe-Name KYC Verification Tier 체계 도입 — 4단계 검증 등급(Tier-0~3) 관리 | Must |
| **REQ-FUNC-056** | Safe-Name 등록 시 최소 Tier-1 KYC 검증 필수 + Verified 배지 부여 | Must |
| **REQ-FUNC-057** | 송금 임계값 기반 KYC 등급 강제 — 고액 송금 시 Tier-2 필수 | Should |
| **REQ-FUNC-058** | Safe-Name Resolve API 응답에 KYC 검증 등급·Verified 배지 포함 | Must |

#### 신규 엔터티 필드 도출

**SAFE_NAME 엔터티 확장 필드:**

| 필드 | 타입 | 제약 | 설명 |
|---|---|---|---|
| **kyc_tier** | string | NOT NULL, DEFAULT 'tier_0' | KYC 검증 등급 (tier_0, tier_1, tier_2, tier_3) |
| **kyc_verified_at** | datetime | NULLABLE | KYC 검증 완료 일시 |

### 3.3 CR-V33-C: 지원되지 않는 온체인 실수 예방

#### 현재 상태 (As-Is)

SRS v0.32는 미지원 체인 검증 요청 시 안내 + 지원 요청을 제공(REQ-FUNC-005)하지만, 이는 Zero-FP Simulate API에 한정된다. Safe-Name 기반 송금에서는 다음 시나리오가 방지되지 않는다.

- 송금자가 Ethereum 자산을 Polygon 전용 수신 주소로 전송 시도
- 수신자의 Safe-Name이 등록된 체인과 송금자의 자산 체인이 불일치하는 경우
- NRM 리졸브 시 외부 네이밍 서비스의 주소가 송금 자산의 체인과 불일치하는 경우

#### MCC와의 핵심 차이점

MCC는 트랜잭션 실행 전에 수신 지갑이 해당 디지털 자산과 블록체인을 지원하는지 확인하고, 비호환 시 송금을 사전 차단하여 자금 손실을 원천 방지한다. 현 SRS에는 이에 해당하는 체인·자산 호환성 사전 검증이 존재하지 않는다.

#### 보완 방향

| 보완 ID | 보완 항목 | 영향 범위 |
|---|---|---|
| CR-V33-C-1 | **Asset-Chain Compatibility Gate** 신설 — Safe-Name 송금 플로우에서 리졸브된 주소의 체인과 송금 자산의 체인을 교차 검증하여, 불일치 시 트랜잭션을 차단 | REQ-FUNC-015 수정, 신규 REQ-FUNC |
| CR-V33-C-2 | Safe-Name 등록 시 **지원 자산·체인 목록(Supported Assets/Chains)**을 함께 등록하도록 확장하여, 수신 가능 자산을 명시 | SAFE_NAME 엔터티 확장, REQ-FUNC-014 수정 |
| CR-V33-C-3 | NRM 리졸브 시 **원본 네이밍 서비스의 체인과 송금 체인 간 호환성 경고**를 표시 — 크로스체인 주소 오류 방지 | REQ-FUNC-035 수정 |
| CR-V33-C-4 | 미지원 체인/자산 전송 시도 시 **"이 수신자는 [자산명]을 [체인명]에서 수신할 수 없습니다"** 안내 + 지원되는 대안 체인·자산 목록을 표시 | 신규 REQ-FUNC |

#### 신규/수정 요구사항 도출

| ID | 요구사항 요약 | Priority |
|---|---|---|
| **REQ-FUNC-059** | Asset-Chain Compatibility Gate — 송금 전 수신자 체인·자산 호환성 사전 검증 및 비호환 시 차단 | Must |
| **REQ-FUNC-060** | Safe-Name 수신 가능 자산·체인 등록 — 등록 시 지원 자산·체인 목록 설정 | Must |
| **REQ-FUNC-061** | NRM 리졸브 시 크로스체인 호환성 경고 표시 | Should |

#### 신규 엔터티 필드 도출

**SAFE_NAME 엔터티 확장 필드:**

| 필드 | 타입 | 제약 | 설명 |
|---|---|---|---|
