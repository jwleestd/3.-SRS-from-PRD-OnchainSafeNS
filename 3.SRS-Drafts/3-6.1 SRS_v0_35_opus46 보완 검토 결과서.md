### 1.1 일회용 주소(Disposable Address)의 보안적 의의

전통 금융에서는 거래마다 고유한 가상계좌번호를 사용하여 수신자의 실계좌를 외부에 노출하지 않는 것이 보안의 기본이다. 온체인 환경에서도 동일한 원리로, 거래마다 일회성 수신 주소를 생성하면 다음의 보안 위협을 방어할 수 있다.

| 위협 | 영구 주소 사용 시 | 일회용 주소 사용 시 |
|---|---|---|
| 주소 추적·프로파일링 | 모든 거래가 단일 주소에 누적 → 자산 규모·거래 패턴 완전 노출 | 거래마다 별개 주소 → 추적 불가, 프라이버시 보호 |
| 주소 재사용 피싱 | 과거 거래 이력의 주소를 사칭한 피싱 공격 | 이미 만료된 주소 → 피싱 대상 소멸 |
| 클립보드 하이재킹 | 탈취된 영구 주소가 장기간 유효 | 일회용 주소는 사용 즉시 무효화 → 피해 창 최소화 |
| 사기 주소 오인 송금 | 영구 주소 변경 없이 사기 등록 시 장기 피해 | 거래 단위 주소 갱신 → 피해 범위 단건 한정 |

## 2. Gap 분석 결과

### GAP-01. 일회용 주소 생성 메커니즘 완전 부재

| 구분 | 상세 |
|---|---|
| **현 상태 (v0.34)** | Safe-Name은 **고정 지갑 주소(`wallet_address`)**에 1:1 매핑. 모든 이체가 동일한 수신 주소로 입금되어 거래 이력이 단일 주소에 누적됨 |
| **보완 기준 요구** | 금융거래 시마다 **일회성 수신 주소**를 생성·사용·폐기 |
| **Gap** | Disposable Address 생성·관리·포워딩·만료의 전체 생명주기가 미정의. 관련 서비스, API, 엔터티, 시퀀스가 없음 |
| **심각도** | **P0 (Critical)** — 보완 기준의 전체 범위에 해당 |

### GAP-02. 일회용 주소 → 영구 주소 자금 포워딩 메커니즘 부재

| 구분 | 상세 |
|---|---|
| **현 상태 (v0.34)** | 송금은 수신자의 영구 주소로 직접 실행. 중간 포워딩 계층이 없음 |
| **보완 기준 요구** | 일회용 주소로 입금된 자금이 수신자의 영구 지갑으로 **자동 포워딩** |
| **Gap** | 자금 포워딩 스마트 컨트랙트 또는 오프체인 릴레이어, 포워딩 수수료·가스비 정책, 포워딩 실패 시 복구 전략이 전무 |
| **심각도** | **P0 (Critical)** — 일회용 주소의 핵심 동작 요소 |

### GAP-03. 일회용 주소 생명주기 관리 부재

| 구분 | 상세 |
|---|---|
| **현 상태 (v0.34)** | Safe-Name의 상태 관리(active/expired/grace_period 등)는 존재하나, 거래 단위 주소의 생성→활성→사용→만료→폐기 생명주기가 없음 |
| **보완 기준 요구** | 일회용 주소의 **생성, 활성, 사용 완료, 만료, 폐기** 상태 전환 관리 |
| **Gap** | 일회용 주소 만료 정책(시간 기반 vs 사용 기반), 미사용 주소 정리(GC), 만료 후 잔금 처리 정책 미정의 |
| **심각도** | **P0 (Critical)** |

### GAP-04. 일회용 주소와 기존 송금 플로우 통합 부재

| 구분 | 상세 |
|---|---|
| **현 상태 (v0.34)** | §3.4.4 송금 플로우에서 리졸브된 영구 주소로 직접 `Chain: 트랜잭션 실행`. 일회용 주소 생성·반환 단계가 없음 |
| **보완 기준 요구** | 기존 "이름 + 금액" 이체 UX를 유지하면서 일회용 주소를 **내부적으로 자동 적용** |
| **Gap** | Pre-Transfer Verification 후 ~ TX 실행 전 구간에 일회용 주소 생성 단계가 삽입되어야 함. 송금자 UX에는 영향 없어야 함(투명 적용) |
| **심각도** | **P1 (High)** — 기존 UX 유지가 핵심 |

### GAP-05. 일회용 주소 데이터 모델·엔터티 부재

| 구분 | 상세 |
|---|---|
| **현 상태 (v0.34)** | DISPOSABLE_ADDRESS 엔터티 없음. TRANSFER_VERIFICATION_LOG에 일회용 주소 참조 필드 없음 |
| **보완 기준 요구** | 일회용 주소별 생성·사용·만료 이력, 포워딩 TX 기록을 관리 |
| **Gap** | 일회용 주소 엔터티, Safe-Name과의 관계, TRANSFER_VERIFICATION_LOG 연동 필드 미정의 |
| **심각도** | **P1 (High)** |

### GAP-06. 일회용 주소 관련 보안·비용 제약사항 부재

| 구분 | 상세 |
|---|---|
| **현 상태 (v0.34)** | 일회용 주소 키 관리, HD Wallet 파생 전략, 포워딩 가스비 부담 주체, 개인키 보안 정책이 전무 |
| **보완 기준 요구** | 보안 강화가 목적이므로 개인키 관리·비용 통제가 필수 |
| **Gap** | HD Wallet(BIP-44) 기반 주소 파생 전략, 키 관리 HSM/KMS 연동, 포워딩 가스비 비용 모델, Simulation Mode 대응이 누락 |
| **심각도** | **P1 (High)** |

## 4. 개선 방향 (v0.35 반영 계획)

### 4.1 아키텍처 전략: HD Wallet 기반 일회용 주소

**채택 방식:** HD Wallet(BIP-44) 파생 기반 일회용 주소. 각 Safe-Name에 HD Wallet Seed를 안전하게 연결하고, 이체 요청 시마다 신규 파생 경로에서 일회용 수신 주소를 자동 생성한다.

| 설계 원칙 | 상세 |
|---|---|
| 투명성 | 송금자 UX("이름 + 금액")에 변경 없음. 일회용 주소 생성·포워딩은 내부 자동 처리 |
| 1회성 보장 | 각 주소는 단일 거래에만 사용. 사용 완료 또는 만료(24시간) 후 즉시 비활성화 |
| 자금 포워딩 | 일회용 주소 입금 감지 → 오프체인 릴레이어가 영구 주소로 자동 포워딩 (Meta-Transaction 방식으로 수신자 가스비 0원) |
| 키 관리 | HD Wallet Seed는 KMS(AWS KMS/HashiCorp Vault)에 저장. 파생 키는 사용 후 메모리에서 즉시 삭제 |
| 비용 | 포워딩 가스비는 플랫폼이 부담하되, L2(Polygon/Arbitrum) 우선 사용으로 건당 $0.01 이하 목표 |

### 4.2 신규 요구사항 요약

| ID | 요약 | Priority |
|---|---|---|
| REQ-FUNC-065 | Disposable Address Service — 이체 요청 시 HD Wallet 기반 일회용 수신 주소 자동 생성 | Must |
| REQ-FUNC-066 | 자금 포워딩 — 일회용 주소 입금 감지 후 영구 주소로 자동 포워딩 + 포워딩 실패 복구 | Must |
| REQ-FUNC-067 | 일회용 주소 생명주기 — 생성→활성→사용 완료→만료(24h)→폐기 상태 관리 | Must |
| REQ-FUNC-068 | 기존 송금 플로우 투명 통합 — 송금자 UX 무변경, 수신자에게 일회용 주소 대신 Safe-Name만 표시 | Must |

## 5. 결론

v0.34 SRS는 Safe-Name Credential 기반 이체, 사기 주소 강제 차단 등 강력한 보안 기능을 갖추고 있으나, 모든 거래가 단일 영구 주소로 수신되는 구조적 한계로 인해 **주소 추적·프로파일링, 주소 재사용 피싱, 클립보드 하이재킹** 등의 위협에 노출되어 있다.

v0.35에서는 HD Wallet 기반 일회용 주소 생성·포워딩·생명주기 관리 전략을 도입하여, 기존 "이름 + 금액" 간소화 UX를 유지하면서 거래 단위 주소 격리를 통한 보안 강화를 달성한다.
