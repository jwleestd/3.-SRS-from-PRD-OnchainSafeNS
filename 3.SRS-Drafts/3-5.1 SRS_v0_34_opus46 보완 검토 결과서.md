## 1. 보완 기준 요약

| 항목 | 보완 기준 내용 |
|---|---|
| 가. 개발 목적 | 고객에게 온체인 환경에서 금융서비스 이용 시 **(1) 사기 주소 여부를 사전 경고**하고 **(2) 지원하지 않는 온체인 네트워크를 선택하는 실수를 방지** |
| 나. 업무 처리 예시 (이체) | 고객 A가 고객 B에게 **이름만 입력**하여 송금 → credential(이름↔주소 매핑) 기반 사기 검증 → **사기 주소 확인 시 이체 중단 + 고객 A 통지** → 정상 시 **송금 완료 후 고객 A·B 양측에 이체 결과 통지** |
| UI/UX | 고객은 이체 시 **"이름"과 "금액"만 입력**하고 이체 실시 |

## 2. Gap 분석 결과

### GAP-01. 이체 UI/UX 간소화 — "이름 + 금액"만 입력

| 구분 | 상세 |
|---|---|
| **현 상태 (v0.33)** | REQ-FUNC-015 및 §3.4.4 시퀀스 다이어그램에서 송금자가 **"수신자 이름 입력 + 송금 자산·체인 선택"** 3가지를 입력해야 송금이 시작됨. 즉 UI에서 체인·자산을 수동 선택하는 단계가 존재 |
| **보완 기준 요구** | 고객은 **"이름"과 "금액"만** 입력. 체인·자산 선택은 시스템이 자동 처리 |
| **Gap** | 체인·자산 **자동 결정(Auto-Select)** 로직이 누락. 수신자의 `supported_chains`/`supported_assets` 정보를 기반으로 시스템이 최적의 체인·자산을 자동 선택하고, 단일 옵션이 아닌 경우에만 선택지를 제시하는 요구사항이 없음 |
| **영향 범위** | REQ-FUNC-015, §3.4.4 시퀀스 다이어그램, Safe-Name Resolve API 응답, Pre-Transfer Verification API |
| **심각도** | **P0 (Critical)** — 보완 기준의 핵심 UI/UX 요구와 직접 충돌 |

### GAP-02. 이체 완료 후 송수신 양측 통지

| 구분 | 상세 |
|---|---|
| **현 상태 (v0.33)** | §3.4.4 시퀀스 다이어그램 마지막 단계: `App-->>Sender: 송금 완료 안내` — **송금자(고객 A)에게만** 완료 안내. 수신자(고객 B)에 대한 이체 결과 통지가 없음 |
| **보완 기준 요구** | 이체 완료 후 **고객 A(송금자)와 고객 B(수신자) 양측**에 이체 결과를 통지 |
| **Gap** | 수신자 앞 이체 완료 통지 플로우 및 관련 기능 요구사항이 완전히 누락. 통합 알림 게이트웨이(Unified Notification Gateway)를 통한 수신자 통지 경로가 정의되지 않음 |
| **영향 범위** | REQ-FUNC-015, §3.4.4 시퀀스 다이어그램, 통합 알림 게이트웨이 연동, TRANSFER_VERIFICATION_LOG 엔터티 |
| **심각도** | **P0 (Critical)** — 보완 기준의 명시적 요구사항 미충족 |

### GAP-03. 사기 주소 확인 시 이체 "중단" 정책

| 구분 | 상세 |
|---|---|
| **현 상태 (v0.33)** | REQ-FUNC-015 및 §3.4.4: 사기 DB 매칭 시 **"경고 표시 + '위험을 인지하고 진행' 체크박스"** — 즉, 사용자가 위험을 인지하면 **진행 가능** |
| **보완 기준 요구** | 사기 주소로 확인되면 **이체 절차를 중단**하고 고객 A에게 이를 통지. "진행" 옵션 없음 |
| **Gap** | 현재의 "경고 + 선택적 진행" 모델과 보완 기준의 **"강제 차단"** 모델이 상충. 사기 주소 검증 단계에서 verification_result가 `blocked_fraud`일 때 트랜잭션 실행을 원천 차단하는 정책과, 그 차단 결과를 송금자에게 통지하는 명시적 요구사항이 필요 |
| **영향 범위** | REQ-FUNC-015, REQ-FUNC-016, REQ-FUNC-053, §3.4.4 시퀀스 다이어그램, TRANSFER_VERIFICATION_LOG |
| **심각도** | **P0 (Critical)** — 보완 기준의 사기 방지 핵심 정책과 직접 충돌 |

### GAP-04. Credential(이름↔주소 매핑) 개념 명시

| 구분 | 상세 |
|---|---|
| **현 상태 (v0.33)** | Safe-Name 등록 시 `human_name ↔ wallet_address` 매핑이 오프체인 DB에 저장되지만, 이를 "credential"이라는 통합 개념으로 명시하지 않음. 이체 업무 처리 절차에서 credential 기반 자동 리졸브·검증의 End-to-End 흐름이 하나의 개념으로 정의되지 않음 |
| **보완 기준 요구** | **기 등록한 credential(이름, 온체인 주소 매핑)**을 기반으로 플랫폼이 사기 검증·송금을 수행 |
| **Gap** | "credential" 용어 정의 및 credential 기반 이체 전체 흐름(등록→리졸브→검증→이체→통지)의 개념적 통합이 누락. 용어집(§1.3)에 credential 정의 추가 필요 |
| **영향 범위** | §1.3 용어집, §1.1 Purpose, REQ-FUNC-014/015 |
| **심각도** | **P1 (High)** — 개념적 정합성 보완 필요 |

### GAP-05. 미지원 네트워크 선택 사전 방지 (UI 레벨)

| 구분 | 상세 |
|---|---|
| **현 상태 (v0.33)** | Asset-Chain Compatibility Gate(REQ-FUNC-059)가 송금 체인·자산과 수신자 지원 체인·자산의 호환성을 **검증 시점**에 확인하고 비호환 시 차단. 그러나 이는 검증 단계에서의 사후 차단이며, 송금자가 미지원 네트워크를 **애초에 선택하지 못하도록** UI에서 사전 방지하는 요구사항은 없음 |
| **보완 기준 요구** | **지원하지 않는 온체인 네트워크를 선택하는 실수를 방지** |
| **Gap** | "이름 + 금액"만 입력하는 UI/UX(GAP-01)에서는 체인을 직접 선택하지 않으므로 자동 결정 시 미지원 체인이 선택될 가능성이 원천적으로 차단됨. 그러나 자동 결정이 불가능한 경우(수신자가 다중 체인 지원 등) 선택지 제시 시 **미지원 체인을 선택 불가(disabled) 처리**하는 명시적 UI 요구사항이 필요 |
| **영향 범위** | REQ-FUNC-059, REQ-FUNC-060, UI/UX 요구사항 신규 |
| **심각도** | **P1 (High)** — 기존 Compatibility Gate로 부분 대응되나, UI 레벨 사전 방지 명시 필요 |

### GAP-06. 이체 결과 통지 내용 명세

| 구분 | 상세 |
|---|---|
| **현 상태 (v0.33)** | 이체 완료 시 "송금 완료 안내"만 기술. 통지에 포함되어야 할 구체적 내용(송금자 이름, 수신자 이름, 금액, 체인, 자산, TX 해시, 시각 등)이 정의되지 않음 |
| **보완 기준 요구** | 고객 A·B에게 **이체 결과를 통지** |
| **Gap** | 이체 결과 통지 메시지의 필수 포함 항목 정의 누락 |
| **영향 범위** | 신규 요구사항 추가, 통합 알림 게이트웨이 연동 |
| **심각도** | **P2 (Medium)** — 기능적으로는 GAP-02에 종속되나 별도 명세 필요 |

## 4. 개선 방향 (v0.34 반영 계획)

### 4.1 요구사항 변경/추가 계획

| 대상 | 개선 내용 |
|---|---|
| **REQ-FUNC-015 변경** | 송금자 입력을 "이름 + 금액"으로 간소화. 체인·자산은 시스템이 수신자 credential 기반 자동 결정. 다중 체인 지원 시에만 선택지 제시(미지원 체인 disabled 처리) |
| **REQ-FUNC-016 변경** | 사기 DB 매칭 시 "경고 + 체크박스" → **"강제 차단 + 송금자 통지"**로 정책 변경. 이체 절차 원천 중단 |
| **REQ-FUNC-053 변경** | Pre-Transfer Verification 결과에 사기 주소 확인 시 강제 차단 정책 반영 |
| **REQ-FUNC-062 신규** | 이체 완료 후 송수신 양측에 통합 알림 게이트웨이를 통한 이체 결과 통지. 통지 내용: 송수신자 이름, 금액, 자산, 체인, TX 해시, 시각 |
| **REQ-FUNC-063 신규** | 체인·자산 자동 결정(Auto-Select) 로직: 수신자 supported_chains/supported_assets 기반 최적 체인·자산 자동 선택. 단일 옵션 시 자동 확정, 다중 옵션 시 선택지 제시(미지원 체인 disabled) |
| **REQ-FUNC-064 신규** | 사기 주소 이체 차단 시 송금자 앞 통지: 차단 사유, 수신자 사기 이력 요약, 신고 안내 포함 |

### 4.2 시퀀스 다이어그램 변경

| 대상 | 개선 내용 |
|---|---|
| **§3.4.4 Safe-Name 기반 송금 플로우** | (1) 송금자 입력을 "이름 + 금액"으로 변경 (2) 체인·자산 자동 결정 단계 추가 (3) 사기 DB 매칭 시 강제 차단 + 통지 분기 (4) 이체 완료 후 송수신 양측 통지 단계 추가 |

### 4.3 용어집·개념 추가

| 대상 | 개선 내용 |
|---|---|
| **§1.3 용어집** | "Safe-Name Credential" 용어 추가 — 유저가 사전 등록한 이름↔온체인 주소 매핑 정보. 이름 기반 이체 시 자동 리졸브·검증의 기반 |
| **§1.1 Purpose** | 개발 목적에 "credential 기반 이름 입력만으로 안전한 이체 수행" 개념 명시 |

| **Safe-Name Resolve API** | 응답에 `recommended_chain`, `recommended_asset` 필드 추가 (자동 결정 추천) |

### 4.5 데이터 모델 변경

| 대상 | 개선 내용 |

## 5. 결론

v0.33 SRS는 KYC Tier·Pre-Transfer Verification·Asset-Chain Compatibility Gate 등을 통해 송수신자 신뢰성 강화의 기술적 기반을 갖추고 있으나, 보완 기준이 요구하는 **"이름 + 금액만으로 이체하는 간소화된 UX"**, **"사기 주소 확인 시 강제 차단"**, **"송수신 양측 이체 결과 통지"** 3개 핵심 요구(P0)가 미반영되어 있다.

v0.34에서는 이 3개 P0 Gap과 3개 P1~P2 Gap을 반영하여, 보완 기준의 개발 목적·업무 처리 예시·UI/UX 요구를 완전히 충족하도록 개선한다.
