# SRS v0.3.1 보완 검토 결과서

**Document ID:** SRS-REVIEW-001  
**Base Document:** SRS-001 v0.3.1  
**검토일:** 2026-05-02  
**검토 기준:** 사용자 제시 6개 보완 기준

## 1. 검토 개요

본 검토서는 SRS v0.3.1 문서를 6개 보완 기준에 대해 Gap 분석하고, 각 기준별 현행 상태(AS-IS)·개선 방향(TO-BE)·영향 범위를 정리한다.

---

## 2. 기준별 Gap 분석

### CR-REVIEW-1: 이종 온체인 네이밍 서비스 미들웨어 전략

| 항목 | 내용 |
|---|---|
| **보완 기준** | Ethereum 외 Polygon, Arbitrum 등 다양한 온체인 네이밍 서비스를 지원하는 전략 명시. 향후 신규 네이밍 서비스 대응을 위한 확장성·일관성을 갖춘 미들웨어 역할 |
| **AS-IS 현황** | ENS(.eth)와 Unstoppable Domains(.crypto/.wallet) 2종만 명시. `ExternalNameResolver`가 `ensAdapter`·`unstoppableAdapter` 2개 어댑터를 하드코딩. Adapter Pattern을 언급하나 체인별 네이밍 확장 전략(SpaceID/BNB, Bonfida/Solana 등) 미기술. 미들웨어 계층 개념 없음 |
| **Gap 요약** | ① 체인별 네이밍 서비스 매핑 전략 부재 ② 신규 네이밍 서비스 플러그인 추가 절차 미정의 ③ 통합 리졸브 미들웨어(Name Resolution Middleware) 아키텍처 미기술 ④ 이름 형식 자동 감지 로직의 확장성 미보장 |
| **영향 섹션** | §1.2 In-Scope IS-8, §3.1 External Systems, §3.2 Client Applications, §3.2.2 Component Diagram, §3.3 API Overview, §4.1 F4-A, §6.2.2 Class Diagram, §6.3.6 시퀀스, 용어(§1.3), REQ-FUNC-035/036, Traceability Matrix |
| **개선 방향** | (가) Name Resolution Middleware(NRM) 계층 신설 — Adapter Registry 기반 플러그인 아키텍처로 신규 네이밍 서비스 무중단 등록 지원 (나) 체인-네이밍 서비스 매핑 테이블 관리 (다) 지원 네이밍 서비스 확대: SpaceID(.bnb), Bonfida(.sol), Lens Protocol(.lens) 등 (라) NRM Adapter SPI(Service Provider Interface) 규격 정의 |

### CR-REVIEW-2: Safe-Name 독자적 이름 등록·관리 및 비용 모델

| 항목 | 내용 |
|---|---|
| **보완 기준** | 가스비 부담 완화를 위해 Safe-Name을 독자적으로 이름 등록·관리. DNS 제도를 참고한 운용 비용 부과 방법 명시 |
| **AS-IS 현황** | Safe-Name 등록은 온체인 레지스트리 컨트랙트에 직접 기록. 메타 트랜잭션(릴레이어 대납)으로 가스비 완화하나, 본질적으로 모든 등록이 온체인 TX 발생. L2 우선 배포(REQ-NF-037)로 비용 절감하나 독자 오프체인 레지스트리 개념 없음. DNS식 연간 갱신 비용 모델 없음 |
| **Gap 요약** | ① 오프체인 우선 레지스트리(Off-Chain First Registry) 전략 부재 — 등록·조회를 오프체인 DB에서 수행하고, 온체인은 주기적 앵커링·증명용으로만 사용하는 구조 미설계 ② DNS식 비용 모델(연간 등록비, 갱신비, 프리미엄 이름 경매 등) 미정의 ③ 도메인 생명주기(등록→갱신→만료→유예→삭제) 관리 프로세스 미기술 ④ 릴레이어 비용 상한($2,000/월)만 있고, 수익으로 상쇄하는 비즈니스 모델 부재 |
| **영향 섹션** | §1.2 IS-4, §3.2.2 Component Diagram, §4.1 F4/F4-B, §4.2 NFR(비용), §6.2 SAFE_NAME 엔터티, §6.2.2 Class Diagram, §6.3.3 시퀀스, CON-15, 용어(§1.3) |
| **개선 방향** | (가) Off-Chain First 레지스트리 아키텍처: 등록·조회는 오프체인 DB, 소유권 증명·분쟁 해결 시에만 온체인 앵커링(Merkle Root) (나) DNS 참조 비용 모델 도입: 연간 등록비(일반 $5/년, 프리미엄 경매), 갱신비, 유예 기간(Grace Period 30일), 삭제 후 재등록 대기(Redemption 30일) (다) 도메인 생명주기 상태머신 정의 (라) 수익-비용 균형 시산표 추가 |

### CR-REVIEW-3: 외부 Webhook 통합 (Slack 외 Gmail, 카카오톡 등)

| 항목 | 내용 |
|---|---|
| **보완 기준** | Slack API만 지원하는 외부 Webhook을 Gmail, 카카오톡 등 다양한 채널로 통합 지원 |
| **AS-IS 현황** | Notification Service가 `SlackClient`·`PagerDutyClient`·`PushNotificationService` 3종 보유. 핫라인 알림·정책 변경 경보 등 모두 Slack 채널로 고정. 이메일(Gmail)·메신저(카카오톡)·SMS 등 채널 다양화 미지원. 유저별 알림 채널 선호(preference) 설정 미존재 |
| **Gap 요약** | ① Slack 단일 채널 의존 — 고객·이용기관·운영기관 각 역할별 적합 채널 미분류 ② 통합 알림 게이트웨이(Unified Notification Gateway) 아키텍처 부재 ③ 알림 채널 어댑터(Email, KakaoTalk, SMS, Microsoft Teams, Discord 등) 플러그인 구조 미정의 ④ 유저/기관별 알림 선호 설정(Notification Preference) 기능 부재 |
| **영향 섹션** | §3.1 External Systems(Slack API), §3.2.2 Component Diagram(SVC7 NotificationService), §4.1 F2(REQ-FUNC-006/008), §4.2.1 REQ-NF-005, §6.2 엔터티(USER 알림 선호 필드), §6.2.2 Class Diagram(NotificationService), 전 시퀀스 다이어그램 내 Slack 참조 |
| **개선 방향** | (가) Unified Notification Gateway 서비스 계층 신설 — Channel Adapter 패턴으로 Slack·Email·KakaoTalk·SMS·Webhook 등 플러그인 방식 채널 관리 (나) 알림 채널 어댑터 Registry 및 SPI 규격 정의 (다) 유저·기관별 Notification Preference 설정 (라) 알림 라우팅 엔진: 중요도·역할·선호에 따른 채널 자동 선택 |

### CR-REVIEW-4: Stakeholders 3계층 분리 및 Client Applications 재구성

| 항목 | 내용 |
|---|---|
| **보완 기준** | Stakeholders를 ① 고객(최종 이용자), ② 이용기관(서비스 제공자·이용자 관리), ③ 운영기관(플랫폼 운영) 3계층으로 구분하고 Client Applications을 각각 분리 |
| **AS-IS 현황** | Stakeholders 테이블에 C1(VASP CISO), C2(일반 유저), C3(TradFi IT팀장), E1(크립토 포비아), N1/N2(배제 타겟) 나열. 역할 구분이 '기관 vs 개인' 수준이며 3계층(고객·이용기관·운영기관) 분류 미적용. Client Applications도 혼재(VASP SDK, 핫라인 대시보드, 신고 플랫폼, Safe-Name 서비스 등이 계층 구분 없이 나열) |
| **Gap 요약** | ① 플랫폼 운영기관(Admin) 역할·페르소나 부재 — 플랫폼 운영자, 컴플라이언스 담당, Data QA팀 등 미정의 ② Client Applications이 이용자 계층과 미매핑 ③ 운영기관용 Admin 콘솔/대시보드 미정의 ④ 이용기관(VASP, TradFi)과 고객(C2, E1)의 서비스 접점이 혼재 |
| **영향 섹션** | §2 Stakeholders, §3.2 Client Applications, §3.2.1 UseCase Diagram, §3.2.2 Component Diagram, 전반적 역할 참조 |
| **개선 방향** | (가) Stakeholders를 3계층으로 재편: Tier-1 고객(C2 일반 유저, E1 크립토 포비아), Tier-2 이용기관(C1 VASP CISO, C3 TradFi IT팀장), Tier-3 운영기관(O1 플랫폼 운영자, O2 컴플라이언스 담당, O3 Data QA팀) (나) Client Applications을 3계층별 그룹화 (다) 운영기관용 Admin Console 신설 (라) UseCase Diagram 3계층 반영 |

### CR-REVIEW-5: 금융결제원 OPEN API 활용으로 변경

| 항목 | 내용 |
|---|---|
| **보완 기준** | 금융결제원 금융공동망 지원 방식을 실시간 자금이체·수신자 확인 전문이 아닌, 금융결제원 OPEN API를 이용하여 사람 이름 또는 법인명 활용으로 변경 |
| **AS-IS 현황** | §3.1에서 금융결제원 금융공동망을 '외부 전문(電文) 인터페이스'로 정의, '실시간 자금이체·수신자 확인 전문 송수신'으로 명시. IS-9에 '금융결제원 금융공동망·ISO 20022 게이트웨이', REQ-FUNC-039에서 '금융공동망 수신자 확인 전문에 매핑', §6.3.7 시퀀스에 KFTC 계좌이체 전문 변환 등 전문 기반 설계가 전면 적용 |
| **Gap 요약** | ① 금융공동망 전문(電文) 인터페이스 → 금융결제원 OPEN API(REST)로 전면 전환 필요 ② 수신자 확인 전문 → OPEN API의 '예금주 조회'·'법인명 조회' API로 변경 ③ Safe-Name ↔ 실명/법인명 매핑·검증 로직 신설 필요 ④ KFTC 전문 포맷터 → OPEN API 호출 클라이언트로 변경 |
| **영향 섹션** | §1.2 IS-9, §1.3 용어(금융결제원, 금융공동망), §1.4 REF-08, §3.1 External Systems(금융결제원), §3.2 Client Applications(금융공동망 게이트웨이), §3.3 API Overview(Financial Gateway API), §4.1 F4-C(REQ-FUNC-039/040), §4.2.9 REQ-NF-038, §6.1 Endpoint A12/A13, §6.2.2 FinancialGatewayService 클래스, §6.3.7 시퀀스, CON-14/15, Traceability Matrix |
| **개선 방향** | (가) 금융결제원 OPEN API(오픈뱅킹 API) 기반 재설계: 예금주 조회 API(계좌번호 → 예금주명), 법인명 조회 API 활용 (나) Safe-Name 등록 시 실명/법인명 KYC 검증 옵션 추가 (다) 기존 전문 포맷터를 OPEN API Client로 대체 (라) ISO 20022 게이트웨이는 유지하되, KFTC 전문 게이트웨이는 OPEN API 게이트웨이로 변경 |

### CR-REVIEW-6: MVP 시뮬레이션 모드 구성

| 항목 | 내용 |
|---|---|
| **보완 기준** | 온체인 사기 방지 플랫폼 SRS를 MVP 시 시뮬레이션으로 동작하도록 구성 |
| **AS-IS 현황** | SRS 전반에 걸쳐 실제 온체인 연동, 실제 외부 API 호출, 실제 스마트 컨트랙트 배포 등을 전제. 시뮬레이션/샌드박스/스텁 모드에 대한 언급 없음. PoC 환경 테스트(CON-1, CON-15)는 제약사항으로만 언급되고, MVP 전체가 시뮬레이션으로 동작하는 아키텍처 미설계 |
| **Gap 요약** | ① MVP Simulation Mode 아키텍처 전략 부재 ② 외부 시스템 스텁(Stub)/목(Mock) 전략 미정의 — RPC 노드, Chainalysis, OFAC, 블록체인 네트워크 등 ③ 테스트넷 vs 시뮬레이터 운영 정책 미기술 ④ 시뮬레이션 데이터 시드(Seed) 전략 미정의 ⑤ 시뮬레이션 ↔ 프로덕션 전환 계획 부재 |
| **영향 섹션** | §1.2 In-Scope 전체, §3.1 External Systems 전체, §3.2.2 Component Diagram, §4.1 전 기능요구사항, §4.2 전 NFR, §6.1 전 Endpoint, §6.2.2 Class Diagram, §6.3 전 시퀀스, §6.4 Validation Plan |
| **개선 방향** | (가) MVP Simulation Mode 아키텍처 섹션 신설: 전 컴포넌트에 Simulation Adapter 계층 삽입 (나) 외부 시스템별 시뮬레이션 전략 정의: Mock RPC(Hardhat/Anvil), Stub Fraud DB(시드 데이터), Testnet(Sepolia/Mumbai/Arbitrum Goerli) (다) 시뮬레이션 데이터 시드 전략: 사기 주소 1,000건, 정상 주소 10,000건, Safe-Name 100건 사전 등록 (라) Simulation → Production 전환 체크리스트 정의 |

## 5. 결론

현행 SRS v0.3.1은 6개 보완 기준 모두에서 Gap이 확인되었다. 특히 MVP 시뮬레이션 모드(CR-6), Stakeholders 3계층 분리(CR-4), Safe-Name 독자 레지스트리(CR-2)는 아키텍처 전반에 영향을 미치므로 우선적으로 반영해야 한다. 개선 결과물은 `3-2.SRS_v0_31_opus46.md`에 반영한다.
