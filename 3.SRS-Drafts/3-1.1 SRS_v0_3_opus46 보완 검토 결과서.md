## 1. 검토 범위 및 보완 기준 요약

본 검토는 아래 3가지 수용 기준에 따라 SRS v0.2의 갭(Gap)을 식별하고 보완 방향을 수립한다.

| # | 보완 기준 | 핵심 키워드 |
|---|---|---|
| CR-1 | 외부 사기 정보 소스(Chainalysis API, OFAC SDN 리스트) 정책 변경 시 우회 조치 방안 명시. 무상 외부 소스 최우선 고려 | 폴백 소스, 무상 우선, 정책 모니터링 |
| CR-2 | 사기 주소 신고·조회 플랫폼에서 고객의 사기 주소 오등록(False Report) 대응 우회 전략 명시 | 이의 신청, 오등록 해제, 피해 방지 |
| CR-3 | Blockchain Networks 네이밍 서비스 활용 시 우회 전략: (가) Safe-Name 연계, (나) 가스비 완화, (다) 금융결제원 금융공동망 + ISO 20022 지원 | 네이밍 서비스 통합, 가스비 최적화, 레거시 금융망 호환 |

## 2. 기준별 갭 분석 및 보완 내역

### 2.1 CR-1: 외부 사기 정보 소스 정책 변경 시 우회 조치

#### 2.1.1 현행(As-Is) 분석

| 위치 | 현행 내용 | 갭 |
|---|---|---|
| CON-7 (§1.2) | "Chainalysis/OFAC API의 외부 개발자 접근 허용 정책이 최소 6개월 유지된다" — 의존성으로 기술 | 정책 변경 **시** 구체적 우회 조치가 부재. 대체 소스 목록·전환 절차·무상 우선 정책 없음 |
| CON-13 (§1.2) | "복수 외부 소스 병렬 확보 + 자체 커뮤니티 신고 DB 1차 소스 육성" — Risk로 기술 | 복수 소스의 구체적 후보(특히 무상 소스)가 명시되지 않음. 전환 트리거·자동화 절차 미정의 |
| REQ-FUNC-024 (§4.1 F6) | Agent가 외부 소스에서 수집·정규화·적재 | 소스 장애/정책 변경 시 대체 소스 자동 전환 요구사항 없음 |
| REQ-FUNC-026/027 (§4.1 F6) | 부분·전면 장애 시 폴백 | 장애 시 폴백은 있으나, **정책 변경에 의한 영구 차단** 시나리오 대응 없음 |
| §3.1 External Systems | Chainalysis, OFAC만 기술 | 무상 대체 소스 미기재 |

#### 2.1.2 보완 방향

1. **CON-7 확장** → 정책 변경 감지 트리거(월 1회 API 변경 로그 자동 모니터링) 및 30일 전환 기한 명시
2. **CON-13 확장** → 무상 외부 소스 우선 순위 목록 구체화 (Etherscan Labels API(무상) > MistTrack(무상 티어) > ScamSniffer(무상) > Chainalysis(유료) > OFAC SDN(공공))
3. **신규 CON-14** → 무상 소스 전환 정책 원칙: "유료 소스 불가 시 무상 소스를 최우선 대체로 채택" 명문화
4. **REQ-FUNC-024 보완** → 소스 정책 변경 시 Agent가 대체 소스로 자동 전환하는 요구사항 추가
5. **신규 REQ-FUNC-031** → 외부 소스 정책 변경 감지 및 자동 전환 기능 요구사항 신설
6. **§3.1 External Systems** → 무상 대체 소스 3종 추가 등록
7. **시퀀스 다이어그램 보완** → Fraud Intelligence Agent 플로우에 정책 변경 감지·전환 경로 추가

#### 2.1.3 영향 범위

| 영향 섹션 | 변경 유형 |
|---|---|
| §1.2 Constraints | CON-7 수정, CON-13 수정, CON-14 신규 |
| §3.1 External Systems | 무상 대체 소스 3종 추가 |
| §4.1 F6 (REQ-FUNC-024) | Acceptance Criteria 보강 |
| §4.1 F6 (REQ-FUNC-031) | 신규 요구사항 추가 |
| §5 Traceability Matrix | REQ-FUNC-031 행 추가 |
| §6.3.2 시퀀스 다이어그램 | 정책 변경 전환 경로 추가 |
| §1.3 Definitions | 무상 소스 관련 용어 추가 |

### 2.2 CR-2: 사기 주소 오등록(False Report) 대응 우회 전략

#### 2.2.1 현행(As-Is) 분석

| 위치 | 현행 내용 | 갭 |
|---|---|---|
| CON-10 (§1.2) | "사기 주소 오등록 — 검증 파이프라인 PoC + 다단계 검증 병행, 오등록률 0.5% 이하" | 오등록 **발생 후** 피해 주소 소유자의 이의 신청·해제 프로세스 미정의 |
| REQ-FUNC-013 (§4.1 F3) | 중복/스팸 신고 차단 | 스팸 차단은 있으나, 검증 완료 후 오등록 판명 시 해제·보상·신고자 페널티 없음 |
| REQ-NF-014 (§4.2.2) | 오등록률 0.5% 이하 | 오등록 검출 후 복구(remediation) 절차·SLA 없음 |
| FRAUD_ADDRESS 엔터티 (§6.2) | status: pending, verified, rejected | "disputed(이의 제기)" 상태 없음 |
| FRAUD_REPORT 엔터티 (§6.2) | status: submitted, verifying, verified, rejected | 오등록 판정 후 "false_report(오신고)" 상태 없음 |

#### 2.2.2 보완 방향

1. **신규 REQ-FUNC-032** → 사기 주소 오등록 이의 신청 기능: 피해 주소 소유자가 "이의 신청" 버튼으로 해제 요청 + 증빙 제출
2. **신규 REQ-FUNC-033** → 이의 심사 및 자동 해제: 이의 심사 SLA <= 48시간, 오등록 확인 시 즉시 해제 + 신고자에게 사유 통보 + 반복 오신고자 페널티(신고 제한)
3. **신규 REQ-FUNC-034** → 오등록 피해 보상 안내: 오등록 확인 시 피해 주소 소유자에게 피해 보상 절차 안내 + 재발 방지 모니터링 등록
4. **CON-10 보강** → 오등록 발생 후 복구 SLA 및 이의 처리 프로세스 명시
5. **FRAUD_ADDRESS 엔터티** → `disputed` 상태 추가
6. **FRAUD_REPORT 엔터티** → `false_report` 상태 추가 + `dispute_count` 필드 추가
7. **신규 엔터티 FRAUD_DISPUTE** → 이의 신청 전용 데이터 모델 신설
8. **시퀀스 다이어그램 신규** → 오등록 이의 신청·심사·해제 플로우

#### 2.2.3 영향 범위

| 영향 섹션 | 변경 유형 |
|---|---|
| §1.2 Constraints (CON-10) | 복구 SLA 보강 |
| §4.1 F3 | REQ-FUNC-032/033/034 신규 |
| §5 Traceability Matrix | 3건 행 추가 |
| §6.2 Entity (FRAUD_ADDRESS) | `disputed` 상태 추가 |
| §6.2 Entity (FRAUD_REPORT) | `false_report` 상태 + `dispute_count` 추가 |
| §6.2 Entity (FRAUD_DISPUTE) | 신규 엔터티 |
| §6.2.1 ERD | 관계 추가 |
| §6.3 시퀀스 다이어그램 | 오등록 이의 플로우 신규 |

### 2.3 CR-3: Blockchain Networks 네이밍 서비스 활용 시 우회 전략

#### 2.3.1 현행(As-Is) 분석

| 위치 | 현행 내용 | 갭 |
|---|---|---|
| §4.1 F4 (REQ-FUNC-014~017) | Safe-Name 등록·리졸브·송금 기능 정의 | 외부 네이밍 서비스(ENS, Unstoppable Domains 등)와의 연계/통합 전략 없음 |
| §3.1 External Systems | Blockchain Networks에 "네이밍 서비스 레지스트리 컨트랙트" 기술 | Safe-Name 외 타 네이밍 서비스 리졸브 연계 미정의 |
| Safe-Name Service (§3.2) | 이름 등록, 이름 기반 송금, 주소 리졸브 | (가) 외부 네이밍 서비스 → Safe-Name 교차 리졸브 미지원 |
| 가스비 | CON-8에서 RPC 비용만 언급 | (나) Safe-Name 등록·변경 시 가스비 부담 완화 전략 없음 |
| 금융 표준 | 미언급 | (다) 금융결제원 금융공동망·ISO 20022 메시지 표준 지원 전무 |

#### 2.3.2 보완 방향

**(가) Safe-Name 서비스와 외부 네이밍 서비스 연계**

1. **신규 REQ-FUNC-035** → 외부 네이밍 서비스(ENS, Unstoppable Domains 등) 이름을 Safe-Name Resolve API에서 교차 리졸브 지원. 외부 네이밍 → 주소 리졸브 후 사기 DB 교차 검증 동일 적용
2. **신규 REQ-FUNC-036** → Safe-Name 미등록 유저가 외부 네이밍 서비스 이름을 보유한 경우, Safe-Name 원클릭 연동(Import) 기능 제공
3. **§3.1 External Systems** → ENS(Ethereum Name Service), Unstoppable Domains API 외부 시스템 추가

**(나) 가스비 부담 완화**

4. **신규 REQ-FUNC-037** → Safe-Name 등록·변경 시 메타 트랜잭션(Meta-Transaction / Gasless Relay) 지원으로 유저 가스비 부담 제거 또는 플랫폼 대납
5. **신규 REQ-FUNC-038** → 배치 등록(Batch Registration) 지원: 다수 이름 일괄 등록 시 가스비 최적화 (단일 트랜잭션 번들링)
6. **신규 REQ-NF-037** → L2 우선 배포 전략: 가스비가 저렴한 L2(Polygon, Arbitrum) 레지스트리를 기본으로 하고, L1(Ethereum)은 선택적 미러링
7. **신규 CON-15** → 메타 트랜잭션 릴레이어(Relayer) 운영 비용이 월 $2,000 이내로 통제 가능하다 — 가정

**(다) 금융결제원 금융공동망 + ISO 20022 지원**

8. **신규 REQ-FUNC-039** → Safe-Name Resolve API가 금융결제원 금융공동망 전문 형식(계좌이체 전문)으로 수신자 정보를 변환·응답할 수 있는 게이트웨이 엔드포인트 제공
9. **신규 REQ-FUNC-040** → Safe-Name 송금 요청·응답 메시지가 ISO 20022(pacs.008, pacs.002) 표준 메시지 구조를 지원하여, 국제 금융 메시징 시스템과 호환
10. **신규 REQ-NF-038** → 금융공동망·ISO 20022 게이트웨이 응답 시간 p95 <= 1,000ms
11. **§1.3 Definitions** → ISO 20022, 금융결제원, Meta-Transaction, ENS 등 용어 추가

#### 2.3.3 영향 범위

| 영향 섹션 | 변경 유형 |
|---|---|
| §1.2 Constraints | CON-15 신규 |
| §1.3 Definitions | 5종 용어 추가 |
| §3.1 External Systems | ENS, Unstoppable Domains, 금융결제원 금융공동망 추가 |
| §3.3 API Overview | 게이트웨이 API 2종 추가 |
| §4.1 F4 | REQ-FUNC-035~040 (6건 신규) |
| §4.2 NFR | REQ-NF-037, REQ-NF-038 (2건 신규) |
| §5 Traceability Matrix | 8건 행 추가 |
| §6.2 Entity (SAFE_NAME) | 외부 네이밍 연계 필드 추가 |
| §6.3 시퀀스 다이어그램 | 외부 네이밍 리졸브·가스비 릴레이·금융망 게이트웨이 플로우 신규 |

## 4. 결론

SRS v0.2는 핵심 기능 요구사항 및 아키텍처가 충실하나, 위 3가지 보완 기준에 대해 총 12건의 신규 요구사항(기능 10건 + 비기능 2건)과 관련 제약·엔터티·시퀀스 보완이 필요하다. 보완된 SRS v0.3은 `3-2.SRS_v0_3_opus46.md`에 반영되었다.
