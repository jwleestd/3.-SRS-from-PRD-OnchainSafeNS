## 1. 검토 개요

본 검토서는 SRS v0.31 문서를 3개 보완 기준에 대해 Gap 분석하고, 각 기준별 현행 상태(AS-IS)·개선 방향(TO-BE)·영향 범위를 정리한다.

### CR-REVIEW-V32-2: 무료 사기 정보 사이트 정기 수집 전략 + 운영 담당자 승인 워크플로우

| 항목 | 내용 |
|---|---|
| **보완 기준** | 다양한 무료 사기 정보 제공 사이트에서 정기적으로 수집하는 전략을 명시. 수집된 사기 정보의 내부 DB 추가·업데이트 시 운영기관 담당자(O3) 승인을 받은 후 수행 |
| **AS-IS 현황** | Etherscan Labels, MistTrack, ScamSniffer 3종 무상 소스를 외부 시스템으로 기재. REQ-FUNC-024에서 "정규화하여 통합 DB에 적재"로만 기술. 수집→DB 적재 사이에 운영기관 담당자의 승인 프로세스 없음. 수집 주기 <= 15분으로만 명시되고, 사이트별 수집 전략(크롤링/API/RSS 등) 미구체화. 수집 데이터의 품질 검증·승인 워크플로우 부재 |
| **Gap 요약** | ① 무료 사기 정보 소스의 정기 수집 전략(소스별 수집 방식·주기·데이터 형식) 미구체화 ② 수집 → 스테이징 → 품질 검증 → 운영 담당자 승인 → 본 DB 반영의 파이프라인 미정의 ③ 승인 워크플로우(O3 Data QA팀 또는 O2 컴플라이언스 담당)를 위한 UI(승인 대기열·일괄 승인·거부 사유) 미존재 ④ 자동 승인 룰(Auto-Approve Rule)과 수동 승인 기준 미분류 |
| **영향 섹션** | §3.1, §4.1 F6(REQ-FUNC-024), F6-A(REQ-FUNC-031), §3.2.3 Tier-3 Client, §6.2.2 Class Diagram(FraudIntelligenceAgent), §6.3.2 시퀀스, FRAUD_ADDRESS/FRAUD_INTELLIGENCE_SOURCE 엔터티, Traceability Matrix |
| **개선 방향** | (가) 무료 소스별 수집 전략 테이블 신설: Etherscan Labels(REST API, 1시간 주기), MistTrack(REST API, 4시간 주기), ScamSniffer(REST API, 2시간 주기), OFAC SDN(공공 다운로드, 일 1회), ChainAbuse(크롤링/API, 일 1회) 등 (나) 수집 파이프라인 3단계 정의: ① Raw 수집 → Staging DB ② 자동 품질 검증(중복 제거, 형식 검증, 기존 DB 교차 확인) ③ 운영 담당자(O3) 승인 → 본 DB 반영 (다) 자동 승인 룰: 외부 3개 소스 이상 교차 확인된 주소는 자동 승인, 단일 소스만 확인된 주소는 수동 승인 대상 (라) 승인 대기열 UI를 Data QA 대시보드에 통합 |

### CR-REVIEW-V32-3: 운영기관용 Client Applications 기능별 재분해 및 Tier-1/2 연계 명확화

| 항목 | 내용 |
|---|---|
| **보완 기준** | 운영기관용 Client Applications이 유사 기능별로 분해가 부족하므로, 고객용·이용기관용 Client Applications 클라이언트와 연계하여 관련 유사 기능별로 모아 명확히 구분. 향후 개발·유지보수가 쉽도록 전략 명시 |
| **AS-IS 현황** | Tier-3 운영기관용 Client는 3개로 분류: ① Admin Console(O1) — 시스템 모니터링, SLA, 비용, 알림 채널, 모드 전환, 이용기관·고객 관리, NRM 어댑터 관리 등 7개 기능 혼재 ② 컴플라이언스 대시보드(O2) — 이의 심사, 오등록 판정, Safe-Name 분쟁, 금융 규제 ③ Data QA 대시보드(O3) — 사기 DB 품질, 소스 커버리지, 데이터 정합성. Admin Console이 과도하게 비대하고, Tier-1/2 Client와의 기능 연계가 명확하지 않음 |
| **Gap 요약** | ① Admin Console(O1)이 7개 이상의 이질적 기능을 하나의 클라이언트에 혼재 — 모니터링, 알림 관리, NRM 관리, 시뮬레이션 전환, 이용기관 관리 등이 단일 앱에 과적재 ② Tier-1 고객용 Client(사기 신고·조회, Safe-Name, Warranty)와 연계되는 운영기관 관리 기능이 분리되지 않음 ③ Tier-2 이용기관용 Client(핫라인, Fraud Agent)와 연계되는 운영기관 관리 기능이 분리되지 않음 ④ 기능별 모듈 분해 전략(도메인 기반)이 부재하여 향후 개발·유지보수 시 변경 영향 범위가 불명확 |
| **영향 섹션** | §3.2.3, §3.2.4 UseCase Diagram, §3.2.5 Component Diagram, §4.1 F9(REQ-FUNC-048), Traceability Matrix |
| **개선 방향** | (가) Admin Console을 도메인 기반 기능 모듈로 분해하여 5개 운영 콘솔로 재구성: ① **사기 정보 관리 콘솔** — Tier-1 사기 신고·조회 플랫폼 및 Tier-2 Fraud Agent 대시보드와 연계. 기능: 사기 DB 수집 승인 워크플로우, 소스 커버리지 모니터링, 데이터 품질 관리 (O3 Data QA + O2 컴플라이언스) ② **Safe-Name 관리 콘솔** — Tier-1 Safe-Name 서비스와 연계. 기능: 이름 분쟁 심사(UDRP), 프리미엄 이름 분류, 비용 모델 설정, NRM 어댑터 관리 (O2 + O1) ③ **핫라인·SLA 관리 콘솔** — Tier-2 B2B 핫라인 대시보드와 연계. 기능: 핫라인 SLA 모니터링, 에스컬레이션 정책 관리, 이용기관 계정 관리 (O1) ④ **Warranty·보증 관리 콘솔** — Tier-1 Warranty 위젯과 연계. 기능: 보증풀 잔고 모니터링, 클레임 심사, 보험사 제휴 관리 (O1 + O2) ⑤ **시스템 운영 콘솔** — 전체 시스템 공통. 기능: 인프라 모니터링(TPS, 가용성, 비용), Simulation/Production 모드 전환, 알림 채널 관리 (O1) (나) 각 콘솔을 독립 배포 가능한 마이크로 프론트엔드(Micro Frontend) 모듈로 설계하여 도메인별 독립 개발·배포·유지보수 가능 (다) 기능-클라이언트 매핑 매트릭스 추가 |

## 5. 결론

현행 SRS v0.31은 3개 보완 기준 모두에서 Gap이 확인되었다. CR-V32-1(KFTC·ISO 삭제)은 다수 섹션에 산재한 참조를 일괄 제거하는 작업이고, CR-V32-2(사기 수집 전략·승인)는 수집 파이프라인에 운영 담당자 승인 단계를 삽입하는 비즈니스 로직 변경이며, CR-V32-3(운영 콘솔 분해)은 Admin Console의 비대한 기능을 도메인 기반 5개 모듈로 분리하는 아키텍처 변경이다. 개선 결과물은 `3-3.2 SRS_v0_32_opus46.md`에 반영한다.
