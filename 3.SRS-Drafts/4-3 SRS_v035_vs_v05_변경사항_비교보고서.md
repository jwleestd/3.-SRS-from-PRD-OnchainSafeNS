## 1. 기술 스택의 명확성

v0.35는 기술 스택을 명시하지 않고 아키텍처 패턴(마이크로서비스, API Gateway 등)만 정의했다. v0.5는 프레임워크·ORM·UI 라이브러리·배포 플랫폼까지 구체적으로 확정하여 바이브코딩 시 AI에게 직접 지시할 수 있는 수준으로 명확화했다.

### 1.1 아키텍처

| 항목 | v0.35 | v0.5 | 변경 사유 |
|---|---|---|---|
| **전체 아키텍처** | 마이크로서비스 (16개 독립 서비스) | **Next.js App Router 단일 풀스택 모놀리스** (C-TEC-001) | 초급 바이브코더가 16개 서비스 간 통신·배포를 관리하는 것이 비현실적 |
| **API 계층** | API Gateway (인증·Rate Limit·라우팅 전담) | **Next.js Middleware + Route Handlers** (C-TEC-002) | 별도 Gateway 서버 없이 단일 앱 내에서 처리 |
| **프론트엔드** | Micro Frontend 5종 (독립 배포) | **단일 Next.js 앱 내 라우트 그룹** (`/(admin)/oc-*`) | Micro Frontend 빌드·배포 파이프라인 제거 |
| **백엔드 프레임워크** | 미지정 (언어·프레임워크 미확정) | **Next.js (TypeScript)** — Server Actions + Route Handlers | 바이브코딩 AI가 가장 잘 생성하는 프레임워크 |

### 1.2 데이터 계층

| 항목 | v0.35 | v0.5 | 변경 사유 |
|---|---|---|---|
| **ORM** | 미지정 | **Prisma ORM** (C-TEC-003) — 스키마 선언적, 타입 안전 | AI 생성 용이, 스키마 기반 마이그레이션 |
| **데이터베이스** | 미지정 (AWS RDS 등 암시) | **SQLite(로컬) / Supabase PostgreSQL(배포)** | Free Tier $0, 초급자 설정 최소화 |
| **캐싱** | Redis (별도 서비스) | **인메모리 캐시 (lib/cache.ts)** | Redis 서버 관리 불필요, Phase-0에서는 캐시 미사용 |
| **메시지 큐** | Kafka/RabbitMQ (암시) | **없음** — 동기 함수 호출 | 비동기 메시징 인프라 제거 |

## 2. MVP 목표 및 가치 전달 조정 내용

v0.35는 71개 FR을 동시에 구현하는 "빅뱅 MVP"였다. v0.5는 Phase 기반 점진적 구현 전략을 도입하여 Phase-0(2~4주)에서 핵심 가치만 검증하고 확장하는 구조로 전환했다.

### 2.1 MVP 범위 및 구현 전략

| 항목 | v0.35 | v0.5 | 변경 사유 |
|---|---|---|---|
| **구현 전략** | 전체 기능 동시 구현 (빅뱅) | **Phase-0/1/2/3 점진적 확장** | 3개월 경험 초급자 바이브코딩 기반 현실적 구현 |
| **Phase 구분** | 없음 (단일 MVP) | **4단계: Phase-0(2~4주) → 1(4~6주) → 2(6~8주) → 3(8~12주)** | 기능별 구현 난이도·의존성 기반 배분 |
| **즉시 구현 범위** | 71 FR / 21 모델 / 25 API | **7 FR(19 REQ) / 6 모델 / 7 API** | 90% 축소하여 실패 리스크 최소화 |
| **개발 기간 목표** | 3~6개월 (추정) | **Phase-0: 2~4주** | MVP의 "빠른 시장 검증" 본래 목적에 부합 |
| **Phase 전환 기준** | 없음 | **§7 Phase Transition Criteria 신설** — E2E 통과, Validation 완료, 비용 확인, 유저 피드백 | 검증 기반 확장 |

### 2.2 Phase-0 핵심 기능 선정

| # | Phase-0 기능 | v0.35 대응 | 선정 사유 |
|---|---|---|---|
