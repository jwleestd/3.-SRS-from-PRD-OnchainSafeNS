## 2. 상세 검토

### AC-1. PRD Story·AC → SRS REQ-FUNC 반영

**판정: ⚠ 부분 충족**

#### 2.1.1 충족 항목

Phase-0 기능 요구사항 7건(REQ-P0-001~019)은 SRS §4.1에 모두 존재한다. Phase-1(§4.3), Phase-2(§4.4), Phase-3(§4.5)의 기능 요구사항 ID도 PRD와 1:1 대응한다.

#### 2.1.2 미충족 항목 (5건)

| # | 이슈 | PRD 위치 | SRS 현황 | 심각도 |
|---|---|---|---|---|
| F-01 | **Phase-0 AC가 GWT 미반영** — PRD는 REQ-P0-001~019 각각에 Given-When-Then + 실패 케이스를 상세 기술하나, SRS는 ≤2초(p95), 정확도 100% 등 요약 지표만 기재 | §5.1.1~5.1.7 | §4.1 | 🔴 높음 |
| F-02 | **FUNC-053~061 번들 미해체** — PRD v0.5는 FUNC-053(KYC Tier-1), FUNC-057(Pre-Transfer), FUNC-059(Compatibility Gate) 3건을 개별 GWT AC로 분리했으나, SRS는 여전히 "FUNC-053~061" 1줄로 번들 | §5.3 | §4.4 | 🔴 높음 |
| F-03 | **Phase-3 FUNC-021~023 번들** — PRD는 잔고 부족(021), 증빙 미충족(022), 수동 폴백(023)을 개별 GWT AC로 기술하나, SRS는 "FUNC-021~023" 1줄로 통합 | §5.4 | §4.5 | 🟡 중간 |
| F-04 | **Phase-3 FUNC-067~069 번들** — 생명주기(067), UX 투명(068), GC(069)도 개별 AC 없이 통합 | §5.4 | §4.5 | 🟡 중간 |
| F-05 | **Phase-2 FUNC-002~005 AC 누락** — PRD에는 FUNC-002(오탐지율 GWT), FUNC-004(RPC 타임아웃 GWT), FUNC-005(미지원 체인 GWT) 개별 AC가 있으나, SRS는 "—"으로 비워둠 | §5.3 | §4.4 | 🟡 중간 |

#### 2.1.3 개선 권고

F-01은 SRS의 핵심 가치인 "구현 가능한 상세 명세" 관점에서 치명적이다. Phase-0 REQ 19건 전체에 PRD의 GWT AC를 SRS에 반영해야 한다. F-02는 PRD v0.5 변경 로그에 명시된 사항이므로 반드시 해체가 필요하다.

### AC-2. KPI·성능 목표 → REQ-NF 반영

**판정: ⚠ 부분 충족**

#### 2.2.1 충족 항목

Phase-0 NFR 10건(REQ-P0-NF-001~010)의 ID, 카테고리, 기준이 PRD §6.1과 1:1 대응한다. Phase-1~3 NFR의 ID(REQ-NF-001~059)와 기준도 대부분 일치한다.

#### 2.2.2 미충족 항목 (4건)

| # | 이슈 | PRD 위치 | SRS 현황 | 심각도 |
|---|---|---|---|---|
| N-01 | **Phase-0 NFR 모니터링·위반 조치 누락** — PRD §6.1은 10건 NFR 각각에 "모니터링 도구"·"위반 시 조치" 컬럼을 명시하나, SRS §4.2는 ID·카테고리·요구사항·기준 4열만 존재 | §6.1 | §4.2 | 🔴 높음 |
| N-02 | **Phase-1~3 NFR 측정 방법·위반 조치 누락** — PRD v0.5에서 추가한 "측정 방법"·"위반 시 조치" 컬럼이 SRS §4.6에 반영되지 않음 | §6.2 | §4.6 | 🔴 높음 |
| N-03 | **추가 NFR 번들** — SRS §4.6 "추가 NFR" 테이블에서 NF-043~044, NF-045~047, NF-049~052, NF-054~056, NF-058~059가 각각 1줄로 번들. PRD는 모두 개별 행으로 기재 | §6.2 추가 NFR | §4.6 | 🟡 중간 |
| N-04 | **PRD KPI Pain ID 역추적 부재** — PRD §1.4의 Pain ID(CORE-1~3, CJM-1~2, EXT-1~2) 목표 KPI가 NFR에 어떻게 매핑되는지 역추적이 없음 | §1.4 | 전체 | 🟡 중간 |

### AC-3. API 목록 → 인터페이스 섹션 반영

**판정: ✅ 충족**

SRS §6.1에 Phase-0 API 7개(P0-A1~A7)와 Phase-1~3 추가 API 19개(A1~A24)가 인증 방식, Rate Limit, Method와 함께 완비되어 있다. PRD §2.2의 "API Route Handlers 7개 / 전체 25개"와 비교하면 SRS가 26개(7+19)로 1개 더 많으나, 이는 PRD 작성 이후 `/transfer/demo` 추가에 따른 자연스러운 증분이다.

### AC-4. 엔터티·스키마 Appendix 완성

**판정: ✅ 충족**

SRS §6.2에 총 22종 엔터티가 Phase별로 완비되어 있다. Phase-0 6종, Phase-1 확장 6종, Phase-2 6종, Phase-3 4종. 각 엔터티에 필드명, 타입, 제약, 설명이 기술되어 있고, Phase-1+ 확장 필드도 별도 테이블로 정리되어 있다. Entity Relationships도 Phase-0과 Phase-1~3으로 나뉘어 텍스트 기반으로 기술되어 있다(ERD 다이어그램은 AC-6에서 별도 평가).
