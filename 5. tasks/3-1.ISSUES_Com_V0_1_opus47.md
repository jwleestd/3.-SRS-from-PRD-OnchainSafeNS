## NFR-103 — Zero-FP 응답 p95 성능 보장

```
title: "[NFR] NFR-103: Zero-FP 응답 p95 ≤ 500ms(Sim) / ≤ 300ms(Prod)"
labels: 'nfr, performance, priority:critical, phase-2'
```

### :dart: Summary
- **기능명:** [NFR-103] Zero-FP 검증 API 응답 성능 보장
- **목적:** VASP의 거래 검증 요청에 대해 Simulation 모드 p95 ≤ 500ms, Production 모드 p95 ≤ 300ms를 일관되게 충족하여 거래 흐름의 지연을 최소화한다.

### :link: References
- SRS 문서: `SRS_v0.8.md#REQ-NF-001`
- 관련 기능: CMD-ZFP-001 (VASP 검증 → Risk Score)

### :white_check_mark: Task Breakdown
- [ ] CMD-ZFP-001 Route Handler에 응답 시간 측정 (`performance.now()`)
- [ ] APM 도구 연동 (Vercel Analytics 또는 Sentry Performance)
- [ ] p50/p95/p99 메트릭 대시보드 구성
- [ ] 성능 저하 알림 (p95 > 임계치 시 Slack)
- [ ] 캐싱 전략 적용: FRAUD_ADDRESS 자주 조회되는 주소 → Redis(Upstash) 5분 TTL
- [ ] DB 쿼리 최적화: 인덱스 활용, N+1 제거
- [ ] 부하 테스트 (k6) 결과 기준 충족 확인
- [ ] 모드별 환경변수: `ZFP_TIMEOUT_SIM_MS=500`, `ZFP_TIMEOUT_PROD_MS=300`

### :test_tube: Acceptance Criteria
**Scenario 1:** Simulation 모드 p95 충족
- Given: SIMULATION_CONFIG.mode='simulation'
- When: 1,000회 검증 요청 부하 테스트
- Then: p95 ≤ 500ms + 에러율 < 0.1%

**Scenario 2:** Production 모드 p95 충족
- Given: SIMULATION_CONFIG.mode='production'
- When: 1,000회 검증 요청 부하 테스트
- Then: p95 ≤ 300ms + 에러율 < 0.1%

**Scenario 3:** 성능 저하 알림
- Given: APM 임계치 설정 (p95 > 500ms 5분 지속)
- When: 임계치 초과
- Then: Slack 경보 채널로 알림 발송

### :gear: Technical Constraints
- REQ-NF-023: Simulation 모드는 외부 RPC 호출 없음 → 응답 시간 단축 우위
- Production 모드는 RPC 호출 포함 → 캐싱 필수
- p95는 월 단위 측정 (출시 후 90일 누적)
- 부하 테스트 환경: 가상 사용자 100명, 30분 지속

### :checkered_flag: Definition of Done
- [ ] APM 모니터링 활성화
- [ ] Simulation 모드 p95 ≤ 500ms 검증
- [ ] Production 모드 p95 ≤ 300ms 검증
- [ ] 알림 자동화 동작 확인

### :construction: Dependencies & Blockers
- **Depends on:** CMD-ZFP-001, CMD-SIM-001
- **Blocks:** None (지속 모니터링 항목)

```
title: "[NFR] NFR-105: 오탐지율 ≤ 0.01% 회귀 테스트 (라벨링 10,000건, 월 1회)"

```

### :dart: Summary
- **기능명:** [NFR-105] 오탐지율 회귀 테스트 체계
- **목적:** 사기 분류 정확도를 라벨링된 10,000건 테스트 세트로 월 1회 회귀 검증하여 오탐지율 ≤ 0.01% (= 1건 이하)를 보장한다.

### :link: References
- SRS 문서: `SRS_v0.8.md#REQ-NF-011`
- 관련 기능: CMD-ZFP-002 (오탐지율 검증 로직)

### :white_check_mark: Task Breakdown
- [ ] 라벨링된 테스트 세트 구축 (10,000건):
  - 정상 주소 (clean) 9,000건 — 검증된 거래소·지갑 주소
  - 사기 주소 (flagged) 1,000건 — OFAC SDN·신고 확정 주소
- [ ] 테스트 자동화 스크립트 작성 (`__tests__/regression/false-positive.test.ts`)
- [ ] 월 1회 Cron 실행 (매월 1일 00:00 UTC)
- [ ] 결과 리포트 생성: 오탐지 건수, 미탐지 건수, 정밀도, 재현율
- [ ] 오탐지율 > 0.01% 시 자동 알림 + 분류 로직 검토 트리거

### :test_tube: Acceptance Criteria
**Scenario 1:** 정상 회귀 통과

- When: 월 1회 회귀 테스트 실행
- Then: 오탐지(false positive) ≤ 1건 → Pass

**Scenario 2:** 회귀 실패 → 알림
- Given: 오탐지 5건 발생
- When: 회귀 테스트 종료

**Scenario 3:** 라벨링 세트 갱신

- When: 라벨링 세트 검토
- Then: 신규 패턴 100건 추가 + 만료 패턴 제거

### :gear: Technical Constraints
- REQ-NF-011: 오탐지율 ≤ 0.01% 의무
- 테스트 세트 크기: 최소 10,000건 (통계적 신뢰도)
- 회귀 주기: 월 1회 (변경 누적분 검증)
- 분기 갱신: 패턴 진화 대응
